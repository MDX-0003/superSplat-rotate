# V4 Pipeline 交接文档 — 2026-06-24

> 本文档记录 `tills/run_pipeline_v4.py` 的全部设计与用法，面向接手开发的同事 / AI Agent。
> v4 是在 v3 基础上为 **flat sequential 图片布局** 重新设计的简化管线。

---

## 一、v4 vs v3 vs v2 对比

| 维度 | v2 (run_pipeline.py) | v3 (run_pipeline_v3.py) | **v4 (run_pipeline_v4.py)** |
|------|---------------------|------------------------|---------------------------|
| 源数据布局 | `raw_frames/{frame:04d}/{camera:03d}.jpg` 两层 | 同 v2 | **flat**: `raw_images/<subdir>/<prefix> {idx:03d}.jpg` |
| 帧索引 | 1-indexed | 1-indexed | **0-indexed**（匹配源文件名） |
| 相机 | 多相机选择 (camera=6) | 多相机选择 (camera=6) | **单相机，camera 字段不读取** |
| COLMAP 依赖 | Step 1: bin→JSON | Step 1: bin→JSON | **删除** |
| UE 依赖 | 无 | ue_seqs/ 检查 | **删除** |
| 圆插值 | Step 3: interpolate_cameras_circle.py | 由 UE Sequence 替代 | **不存在**（v4 只用实拍+渲染） |
| 训练图抽取 | 无 | 无 | **Step 1.5: → Train_imgs/** |
| 段结构 | head+renders+tail | timeline JSON 任意交错 | **real→render→real 三段** |
| 渲染帧数 | 从 config 指定 | 从 JSON 的 start/end 推导 | **全量读取 MP4，不配置帧数** |

## 二、数据布局

### 2.1 Flat 图片格式

```
CameraData/<project>/
  raw_images/                          ← source 字段指向
    DJD-2026-06-23-153925/             ← 自动发现首个子目录
      DJD-2026-06-23-153925 000.jpg    ← prefix + 空格 + idx:03d
      DJD-2026-06-23-153925 001.jpg
      ...
      DJD-2026-06-23-153925 262.jpg    ← 263 张 (000~262)
```

**自动发现逻辑**（`discover_source`）：
1. 先尝试 `source_dir` 根目录直接找 `.jpg`
2. 无文件则取第一个子目录
3. 用最低序号文件的正则 `^(.*?)\s*(\d+)$` 提取 prefix + padding

### 2.2 图像分段（以 CameraData/07 为例）

```
idx 000-074  →  75 张 实拍 (real #1)
idx 074-187  → 114 张 3DGS 训练素材 (被 render 段替代)
                 └─ 抽出到 Train_imgs/<date>/001~114.jpg
idx 188-262  →  75 张 实拍 (real #2)
```

> **注意**：训练范围起始与 real #1 的 end 重叠一位 (idx=74)。这是设计意图：`train_start = prev_real.end`。

## 三、Timeline JSON 规范

### 3.1 Schema

```json
{
  "fps": 25,
  "crf": 6,
  "resolution": "3840x2160",
  "source": "raw_images",
  "segments": [
    { "type": "real",   "start": 0,   "end": 74 },
    { "type": "render", "seq": "seq_f075", "replace_frames": 114 },
    { "type": "real",   "start": 188, "end": 262 }
  ]
}
```

### 3.2 字段说明

| 字段 | 层级 | 类型 | 必需 | 说明 |
|------|------|------|------|------|
| `fps` | 顶层 | int | 否 (默认25) | 输出帧率 |
| `crf` | 顶层 | int | 否 (默认6) | H.264 质量 (越小越好) |
| `resolution` | 顶层 | str | 否 (默认 3840x2160) | 仅记录用途 |
| `source` | 顶层 | str | 否 (默认 raw_images) | 源图目录名 |
| `type` | 段 | "real" \| "render" | **是** | 段类型 |
| `start` | real 段 | int | **是** | **0-indexed** 起始帧索引（含） |
| `end` | real 段 | int | **是** | **0-indexed** 结束帧索引（含） |
| `camera` | real 段 | int | **否 (忽略)** | 向后兼容 v2/v3，v4 不读取 |
| `seq` | render 段 | str | **是** | 渲染 MP4 名称（无 .mp4 后缀） |
| `replace_frames` | render 段 | int | 否 (建议填) | 被替换的源帧数，用于 Step 1.5 训练图抽取 |

### 3.3 约束

- 段顺序必须是 **real → render → real**（恰好三段）
- `start ≤ end`（管道会校验）
- `replace_frames > 0`（有值才触发训练图抽取）
- 源图必须从 0 开始连续编号（中断处 `break` 停止）

## 四、Pipeline 步骤详解

### Step 0: 源数据发现 `discover_source()`

- 输入：`proj`、`source` 配置
- 输出：`(source_dir, prefix, padding)`
- 自动检测文件名前缀与补零宽度
- 支持 root 目录或子目录两种结构

### Step 1.5: 训练图抽取 `extract_train_images()` **[v4 新增]**

- 从 render 段的 `replace_frames` + 前一个 real 段的 `end` 推导范围
- `train_start = prev_real.end`
- `train_end = train_start + replace_frames - 1`
- 目标路径：`Train_imgs/<date_from_prefix>/`
- 输出命名：`001.jpg` 起始，3 位补零，连续序号
- `--force` 会清除已存在的目标目录

**示例**（CameraData/07）：
```
源: DJD-2026-06-23-153925 074.jpg ~ 187.jpg (114 张)
  → Train_imgs/2026-06-23-153925/001.jpg ~ 114.jpg
```

### Step 2: 实拍帧抽取 `extract_real_frames()`

- `out_idx = 0` — **0-indexed 输出命名**
- 源路径：`source_dir / f"{prefix}{idx:0{padding}d}.jpg"`
- 遇到缺失立即 `break`（取 min(config, available)）
- 输出：`anchor_frames/0000.jpg`, `0001.jpg`, ...
- 返回 `real_ranges: dict[seg_idx → (out_start, out_end)]`

### Step 3: 等待渲染 MP4 `wait_for_files()`

- 检查 `renders/<seq>.mp4` 是否存在
- 缺失时打印手动操作提示，`input()` 阻塞等待
- **不再检查 `ue_seqs/`**（v3 残留已删）

### Step 4: 实拍 JPG → MP4 `real_jpgs_to_mp4()`

- 连续 real 段合并为一个 MP4（继承 v3 逻辑）
- ffmpeg：`-start_number {first_out}` where `first_out=0`
- 输入 pattern：`anchor_frames/%04d.jpg`
- 输出命名：`real_{first_src:04d}_{last_src:04d}.mp4`
- 示例：`real_0000_0074.mp4`（75 帧）、`real_0188_0262.mp4`（75 帧）

### Step 5: TS concat → output.mp4 `concat_to_output()`

- **与 v3 完全相同的逻辑**
- 所有 MP4 → TS（`-c copy -bsf:v h264_mp4toannexb`）
- 按 segment 顺序 `concat:` 拼接
- `--debug`：保留中间文件 + 导出 PNG 帧序列

## 七、0-indexed 设计要点

v4 全链路 0-indexed，与 v3 的 1-indexed 彻底分离：

| 项目 | v3 | v4 |
|------|----|----|
| `out_idx` 初始值 | `1` | `0` |
| 首个 anchor 文件 | `0001.jpg` | `0000.jpg` |
| ffmpeg `-start_number` | `1` | `0` |
| Timeline JSON `start`/`end` | 1-indexed | **0-indexed** |
| real MP4 命名 | `real_0001_0074.mp4` | `real_0000_0074.mp4` |
| train img 命名 | 无 | `001.jpg` 起始 (1-indexed，独立计数器) |

> **注意**：train image 输出是 1-indexed (`001.jpg`)，因为 COLMAP 等 3DGS 工具通常期望从 1 开始的文件名。这是 **故意的例外**。

## 十、已知限制

- [ ] Step 3 (SuperSplat 渲染) 仍需手动操作，无法全自动
- [ ] 仅支持恰好 3 段（real→render→real），不支持更复杂的交错
- [ ] 所有 real 段必须来自同一 flat 源（同一 prefix + padding）
- [ ] `replace_frames` 推导训练范围依赖前一个 real 段的 `end`，如果 real 段 `end` 与实际 gap 不对齐，会产生偏移
- [ ] 不支持多 render 段（当前每个项目只有一个 3DGS 重建区间）
