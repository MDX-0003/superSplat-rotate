# 局域网基础：WiFi / 交换机 / IP / SCP 速度

> 汇总自 v7 分布式训练讨论 | 2026-07-03

---

## 1. 基础知识（陈述）

### 1.1 WiFi

WiFi 的物理介质是无线电波（2.4GHz 或 5GHz 频段）。AP（Access Point，即热点/路由器）负责将无线信号与有线网络互转。

- WiFi 是**共享信道**：同一信道上的所有设备（手机、电脑、其他热点）共用一个频段，彼此竞争发送时间。
- WiFi 的**实际速率远低于理论速率**：802.11ac 理论值 866Mbps，实际受距离、墙体、干扰影响，通常只能达到 20-100Mbps。
- WiFi 上行为**半双工**：同一时刻只能发送或接收，不能同时收发。
- WiFi 速率**非对称**：上行和下行速率可以不同。信号弱的设备上行速率更低。

### 1.2 网线 + 交换机

交换机是**二层（数据链路层）设备**，工作单位为 MAC 帧，不处理 IP 层逻辑。

- 交换机**不具备 DHCP 功能，不分配 IP 地址**。
- 交换机**不提供互联网连接**。它只负责把插在上面的设备互相连通。
- 交换机通过**MAC 地址表**转发帧：收到一帧后查目标 MAC 在哪个端口，直接转发到该端口。
- 交换机上同一子网内设备之间的通信**不经过路由器/热点**，全程在交换机的端口间完成。
- 千兆以太网为**全双工**：收发可同时进行，独占信道，不存在无线干扰。

### 1.3 DHCP 和 IP 分配

DHCP（Dynamic Host Configuration Protocol）是设备自动获取 IP 地址的机制。设备启动时发送 DHCP 请求广播，网络中的 DHCP 服务器回复 IP、子网掩码、网关等信息。

- 手机热点内置 DHCP 服务器，会给所有连接的设备分配 IP。
- DHCP 广播帧可以**穿透交换机**到达热点，因此通过交换机连接的设备也能获得 IP。

### 1.4 SCP 传输

SCP（Secure Copy Protocol）基于 SSH 协议传输文件。每个 `scp` 调用建立一次 SSH 连接，在连接内传输文件。

- SCP 的加密/解密开销在现代 CPU 上不是瓶颈（单向加解密可达 100+ MB/s）。
- 小文件（< 1MB）场景下瓶颈是每个文件的 SSH 协议握手（open/write/close 往返），而非带宽。
- SCP 支持**多个源路径**：`scp -r dir1 dir2 file1 user@ip:dst/` 在一次连接内传输所有源。

### 1.5 Windows 多网卡

Windows 可同时启用多张网卡（WiFi + 以太网），每张网卡各自获得一个 IP。

- `ipconfig` 显示所有网卡及其 IP，需注意区分"无线局域网适配器 WLAN"和"以太网适配器"。
- 当两张网卡均能到达同一目标时，Windows 使用**跃点数（Metric）**决定走哪张网卡。跃点数低的优先。
- 如需确保数据走以太网，应**禁用 WiFi 适配器**，仅保留一张网卡。

## 2. 数据传输的两个阶段

以下以"主机通过 SCP 向副机发送文件"为例。

### 第一阶段：地址解析（ARP）

主机知道副机的 IP（如 172.28.52.135），但不知道副机的 MAC 地址。以太网帧需要 MAC 地址才能发送。

1. 主机检查目标 IP 是否在同一子网内（根据子网掩码）。
2. 主机发送 ARP 广播："IP 为 172.28.52.135 的设备，你的 MAC 地址是什么？"
3. 副机收到广播后回复自己的 MAC 地址。
4. 主机在 ARP 表中记录：`172.28.52.135 ↔ MAC XX:XX:XX ↔ 出口网卡：以太网/WiFi`。

### 第二阶段：数据传输（MAC 转发）

主机将数据封装为以太网帧，帧头包含目标 MAC 地址，从 ARP 表记录的出口网卡发出。

- 若走 **WiFi**：帧 → 无线网卡 → 无线电波 → 热点 → 无线电波 → 副机 WiFi 网卡。
- 若走**以太网**：帧 → 以太网卡 → 网线 → 交换机查 MAC 表 → 对应端口 → 网线 → 副机以太网卡。

IP 地址在全程中保持不变，仅用于标识目标。物理路径由 MAC 地址和出口网卡决定。

## 3. 疑难问答

### 交换机不具备 DHCP 功能，为什么通过交换机连接的设备仍然能获得 IP？

DHCP 请求是以太网广播帧（目标 MAC 为 `FF:FF:FF:FF:FF:FF`）。交换机会将广播帧转发到所有端口（除接收端口外），因此 DHCP 广播能穿透交换机到达热点。热点收到请求后回复 IP，回复帧同样通过交换机转发回请求设备。

### IP 来自热点，为什么 SCP 传输不走热点而走交换机？

IP 只在第一阶段（ARP 解析）起作用——确定目标的 MAC 地址。第二阶段实际传输时，以太网帧使用 MAC 地址直接寻址。交换机看到目标 MAC 在本地端口上，直接转发，不经过热点。热点不参与两设备间的数据转发。

### WiFi 模式下 SCP 为什么慢？交换机模式下为什么快？

两个原因：
1. **物理介质不同**：WiFi 是共享无线电波，存在信号衰减、干扰、信道竞争。网线是独占铜线，无干扰、全双工。
2. **转发路径不同**：WiFi 模式下数据经过"发送方→热点→接收方"两次无线传输。交换机模式下数据只经过"发送方→交换机→接收方"两次有线传输，热点不参与。

### 为什么从副机拉取数据（SCP get）比推送数据（SCP put）更慢？

在 WiFi 环境下，SCP get 是副机在发送（上行），SCP put 是主机在发送。两台设备的 WiFi 信号通常不对称——信号弱的设备上行带宽更低。实测数据中：

| 方向 | 发送方 | 速率 |
|------|--------|------|
| PLY 推送到副机 | 主机（信号较好）| 3.8 MB/s |
| PLY 从副机拉取 | 副机（信号较差）| 2.4 MB/s |

### 同时启用 WiFi 和以太网，两个 IP 都有效，SCP 走哪张网卡？

由 Windows 的跃点数（Metric）决定。两张网卡在同一子网时，跃点数低的网卡胜出。但跃点数相同时行为不确定。

**最佳实践**：在交换机环境下**关闭 WiFi 适配器**，仅保留以太网，确保所有流量明确走有线路径。

### 单次 SCP 传多个文件相对于多次 SCP，省掉了哪部分时间？

每次 SCP 调用都要建立一次 SSH 连接（TCP 三次握手 + SSH 密钥协商），约耗时 1-2 秒。`scp -r dir1 dir2 dir3 user@ip:dst/` 在单一 SSH 连接内传输全部数据，只需要一次握手。N 帧合并为一次传输，省掉了 (N-1) 次 SSH 连接建立的开销。

## 4. v7 中的实际影响

| 阶段 | 传输内容 | 当前优化 |
|------|---------|---------|
| Phase 2 分发 | 帧目录（120 张图, ~23MB）→ 各 Worker | `_scp_send_multi` 合并同一 Worker 的全部帧到一次 SCP |
| Phase 4 回收 | PLY 文件（~70MB）← 各 Worker | `_scp_recv_multi` 合并同一 Worker 的全部 PLY 到一次 SCP |
| Phase 2/4 整体 | 多个 Worker 同时传输 | `ThreadPoolExecutor` 并行化各 Worker 的传输线程 |

WiFi 环境下的实测 SCP 速率为 2.2-3.8 MB/s。千兆有线交换机环境下预计可达 60-80 MB/s，传输不再是瓶颈。
